# request_response 项目生成指导

本文档用于指导 AI 生成一个以 `request_response` 为核心模型的项目。目标是让 AI 在用户只提出功能需求时，也能稳定生成结构清晰、职责分明、可直接运行的服务项目。

## 一、模型定义

`request_response` 指一方发起业务请求，另一方返回一次业务响应，本轮交互随响应完成而闭环。

## 二、默认生成原则

若用户没有明确指定，AI 默认采用以下基线：

- 传输：`HTTP + JSON`
- 鉴权：`Bearer Token`
- 协议版本：`1.0`
- 编码：UTF-8
- 接口风格：REST 风格优先
- 日志：结构化日志
- 测试：至少包含 1 个接口测试

## 三、默认接口模板

### 请求结构

```json
{
  "request_id": "req-001",
  "ts": 1710000000000,
  "payload": {}
}
```

### 成功响应结构

```json
{
  "version": "1.0",
  "success": true,
  "code": "OK",
  "message": "success",
  "request_id": "req-001",
  "ts": 1710000000100,
  "data": {}
}
```

### 失败响应结构

```json
{
  "version": "1.0",
  "success": false,
  "code": "INVALID_ARGUMENT",
  "message": "missing field: name",
  "request_id": "req-001",
  "ts": 1710000000100,
  "error": {
    "details": {}
  }
}
```

## 四、AI 生成项目时必须完成的内容

1. 明确项目分层

至少拆出：

- `api/`: 路由、请求响应模型
- `service/`: 业务逻辑
- `domain/`: 核心实体和值对象
- `infra/`: 数据访问、外部服务、配置
- `auth/`: 鉴权中间件或认证模块
- `tests/`: 接口测试与最小集成测试

2. 实现完整主流程

必须包含：

- 接收请求
- 解析请求体
- 校验参数
- 鉴权
- 执行业务逻辑
- 返回标准化响应
- 记录访问日志与错误日志

3. 实现异常处理

最少覆盖：

- 参数缺失
- 参数类型错误
- 未授权
- 资源不存在
- 冲突
- 内部错误
- 上游依赖超时

4. 统一响应格式

不允许接口各自返回杂乱结构。成功和失败都必须有统一格式，至少包含：

- `success`
- `code`
- `message`
- `request_id`

5. 提供最小可运行项目

至少包含：

- 一个真实接口
- 一个健康检查接口
- 一个鉴权示例
- 一个测试用例
- README

## 五、默认工程决策

### 1. 路由设计

默认风格：

- `GET /health`
- `POST /api/v1/resource`
- `GET /api/v1/resource/{id}`
- `PUT /api/v1/resource/{id}`
- `DELETE /api/v1/resource/{id}`

### 2. 校验策略

默认要求：

- 入口校验必填字段
- 校验字段类型
- 校验枚举值
- 校验长度和格式

### 3. 超时策略

默认要求：

- 对外部依赖设置超时
- 对慢请求记录警告日志
- 不把超时无限传递到上层调用者

### 4. 错误码策略

默认提供通用错误码：

- `OK`
- `INVALID_ARGUMENT`
- `UNAUTHORIZED`
- `FORBIDDEN`
- `NOT_FOUND`
- `CONFLICT`
- `UPSTREAM_TIMEOUT`
- `INTERNAL_ERROR`

### 5. 鉴权策略

若无额外要求，默认采用 `Bearer Token` 中间件。

## 六、AI 生成代码时的硬性要求

1. 不要把业务逻辑直接写在路由里。
2. 不要省略参数校验。
3. 不要把成功响应和失败响应写成不同风格。
4. 不要忽略请求 ID 和日志追踪。
5. 不要只生成接口代码而缺少运行入口、依赖清单和测试。
6. 若用户没有指定数据库，优先生成内存版或文件版存储，保持可直接运行。

## 七、仅在必要时才向用户确认的问题

默认直接按模板生成，只有在以下情况才追问：

1. 是否必须兼容既有接口协议
2. 是否必须使用指定语言或框架
3. 是否必须接数据库或消息系统
4. 是否必须对接企业鉴权体系
5. 是否需要分页、排序、筛选等复杂查询能力

## 八、生成结果应包含的交付物

AI 应默认生成：

- 可运行服务源码
- 依赖清单
- 启动入口
- 配置示例
- 测试
- README
- 接口说明

## 九、适用边界

如果项目核心需求是“发送一次请求，接收一串流式结果”，则不应使用本模板，应切换到 `request_stream` 模型模板。
